# 公众号写作风格方法论

2026-05-08 通过深度分析「逛逛GitHub」爆款套路、对比 [[ai-zhiguanju]] 现有文章差距后总结的公众号写作风格方法论。

## 核心原则

### 1. 用真实口吻，不要 AI 腔

- 用自己的真实语气写文章，读者能感受到「真人」
- 避免 AI 常见套话：通用的"值得注意的是"、"让我们来看看"
- 追求自然、有温度的表达

### 2. 一句一段

- 每段只放 1-2 句话
- 手机阅读极舒适，减少视觉负担
- 大量留白，不堆砌文字
- 这是手机端阅读体验的**结构性要求**，不是可选项

### 3. 证据型截图，而非装饰型配图

- GitHub Issue 截图 → 证明争议真实存在
- 代码截图 → 展示项目功能
- 数据截图 → Stars 增长曲线
- **核心原则**：截图是用来「作证」的，不是用来「装饰」的
- 装饰型图片（风景、抽象图）在公众号中价值为零

### 4. 故意呈现两极化观点引发评论

- 不回避负面信息，反而将其作为传播点
- 评论区争议 = 免费传播
- 公众号算法对有互动的文章加权

## 「逛逛GitHub」爆款套路深度拆解

### 标题公式

**数字 + 年龄 + 冲突 + 悬念**

- 「这个GitHub项目被骂上热搜，却拿了13000颗星」
- 制造认知冲突，激发点击欲
- 标题平淡 = 没人点开，内容再好也白搭

### 内容本质

- **本质就是推荐 GitHub 开源项目 + 贴截图，没有实测**
- 不是深度评测，而是信息筛选 + 证据展示
- 降低了内容生产门槛，但阅读体验极佳

### 对比差距分析：AI智管局 vs 逛逛GitHub

| 维度 | AI智管局（旧版） | 逛逛GitHub |
|------|------------------|-------------|
| 标题 | 平淡无奇 | 冲突+数字+悬念 |
| 段落 | 太长（3-5句一段） | 一句一段 |
| 节奏 | 慢、铺垫多 | 快、直入主题 |
| 图片 | 装饰性配图 | 证据型截图 |
| 语气 | AI腔/技术腔 | 真人口吻 |
| 互动 | 无争议点 | 故意引争议 |

### 文章演进

1. **第一版**：「我测了6款免费AI写文章工具」— 传统排版，段落过长
2. **第二版**：加入5张 HTML + Playwright 截图配图 — 开始走向证据型，但排版仍需优化
3. **目标风格**：一句一段 + 真人口吻 + 证据型截图 + 冲突标题

## 写作检查清单

- [ ] 标题包含数字/冲突/悬念至少一种元素
- [ ] 每段不超过2句
- [ ] 图片都是证据型（截图证明某个观点）
- [ ] 通读一遍，不像AI写的
- [ ] 至少有一个争议点或不同观点

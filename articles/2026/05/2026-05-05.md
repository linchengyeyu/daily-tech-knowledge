## 一、gpt2api：把你的ChatGPT账号变成OpenAI API

你有没有想过，自己已经付费的ChatGPT Plus账号，能不能通过API方式调用？

**gpt2api** 就是干这个的——它把你在浏览器里登录的ChatGPT/Grok账户，包装成一个标准的OpenAI兼容API网关。

项目地址：GitHub 432539/gpt2api

**技术栈**：Go + React + MySQL + Redis，Docker Compose 一键部署。

部署完成后你会得到三个端口：
- 17080：用户前端界面
- 17088：管理后台
- 17200：**OpenAI兼容API**（重点！任何支持OpenAI API的工具都能直接接进来）

**实际用途**：搭配 Open WebUI、Hermes Agent 等前端工具，用 ChatGPT Web 账号的白嫖额度来跑 Agent 任务。或者把多个ChatGPT账号做成负载均衡的API池。

**获取Token的方法**：浏览器 F12 → Application → Cookies → 复制 `__Secure-next-auth.session-token` 的值。

## 二、Docker Hub 国内镜像源配置踩坑

部署 gpt2api 的过程中，最大的坑不是项目本身，而是——**Docker Hub 拉不动镜像**。

这个问题在国内太常见了，总结一下实测经验：

**解决方案：配置多个镜像加速源**

在 `/etc/docker/daemon.json` 里加：

```json
{
  "registry-mirrors": [
    "https://docker.1ms.run",
    "https://docker.xuanyuan.me",
    "https://docker.rainbond.cc"
  ]
}
```

注意：单个镜像源不稳定，**必须配多个**。另外镜像源随时可能失效，需要定期更新。

**OrbStack 用户注意**：OrbStack 的 Docker daemon 进程**不走系统 VPN 隧道**，即使你开了代理：
- `docker pull` → daemon 发起请求 → 不走代理 → 直连失败
- `docker run curl google.com` → 容器内进程 → 走宿主网络 → 继承代理 → 成功

所以用了 OrbStack + 代理的同学，老老实实配镜像源吧，别指望代理了。

## 三、Open WebUI 新踩坑：手动创建用户必须插两张表

之前分享过 Open WebUI 连接 Hermes Agent 的部署教程，这次又发现一个新坑：

**手动在 SQLite 里创建账户时，必须同时在 `auth` 表和 `user` 表都插入记录。** 只插 `auth` 表会返回 HTTP 400。

另外还发现 Open WebUI 的一个架构限制：**它不支持显示 Agent 的中间工具调用过程**。你只能看到最终的文本输出，看不到 Agent 调了什么工具、传了什么参数。想看完整过程还是得用命令行或者 Hermes 的 TUI 界面。

如果遇到间歇性断连，可以在 Docker 启动时加两个环境变量：
- `ENABLE_WEBSOCKET_SUPPORT=true`
- `WEBSOCKET_RECONNECT_TIMEOUT=60`

## 🔧 DeepSeek Fallback 400 错误：一个端点引发的血案

事情是这样的：GLM-5.1 触发了限流，系统自动 fallback 到 DeepSeek V4 Flash，结果直接返回 400 错误，整个请求链路断了。

排查后发现，**根因有两个**：

1. **API 端点协议搞错了**。代码里用的是 Anthropic 协议的端点去调 DeepSeek，人家根本不吃这一套。DeepSeek 的 API 文档（api-docs.deepseek.com）写得很清楚，得走 OpenAI 兼容协议。

2. **源码里偷偷 strip 了 thinking blocks**。adapter 层在处理响应时，把 DeepSeek 返回的思考过程块（thinking blocks）给剥离了，导致格式解析直接崩掉。

**修复方案**：切换到 OpenAI 协议端点 + patch adapter 代码，保留 thinking blocks 的处理逻辑。

> 💬 一句话总结：**用别人的 API，先看文档再看协议，别想当然。** fallback 链路不仅要测"能不能调通"，还得测"协议对不对"。

## 🎬 B站评论区的 Shadow DOM 之墙

想在 B站评论区做点自动化操作？你会碰到一个硬茬——`<bili-comments>` 这个 Web Component 用了 **Shadow DOM** 封装，普通的 DOM 操作（querySelector、getElementById）全部失效，根本摸不到里面的元素。

**破局思路：别跟 DOM 死磕，直接调 API。**

用 **OpenCLI**（npm 包名 `@jackwener/opencli`）可以直接调用 B站的评论接口。关键一步是从浏览器的 Cookies 里拿到 CSRF token，然后就可以绕过 Shadow DOM，直接跟后端交互。

操作流程大概是这样：
1. 从 Cookie 中提取 CSRF token
2. 通过 OpenCLI 构造请求，直接调用 B站评论 API
3. 完美绕过 Shadow DOM 的限制

> 💬 一句话总结：**Shadow DOM 是前端封装的利器，但 API 才是终极答案。** 打不过就绕过。

## 🤖 飞书机器人的「静默陷阱」

这个坑真的绝了：飞书机器人配置完成，状态显示"已连接"，一切看起来都很正常。但发消息过去——**石沉大海，一个字都没收到**。

翻了半天日志，最后发现罪魁祸首是两个环境变量的组合拳：

- `FEISHU_ALLOWED_USERS` 为空（没有配置白名单用户）
- `ALLOW_ALL` 为 false（不允许所有人）

结果就是：**机器人连接正常，但谁的消息都不处理，而且没有任何错误提示。**

这种"静默失败"才是最可怕的——不报错、不告警、不提示，就默默地把所有消息吞了。排障的时候你甚至会怀疑是不是网络问题。

**修复方案**：要么在 `FEISHU_ALLOWED_USERS` 中配置允许的用户 ID，要么在开发/测试阶段设置 `ALLOW_ALL=true`。

> 💬 一句话总结：**最可怕的 Bug 不是报错，是静默失败。** 配置项之间的隐式依赖关系，一定要写进文档。

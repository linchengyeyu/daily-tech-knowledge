## 🛠️ OpenCLI：一个CLI搞定90+网站，零Token消耗

先说昨天最大的收获——**OpenCLI**（GitHub: jackwener/opencli）。

你有没有遇到过这种情况：想让AI帮你查B站评论、刷知乎热榜、看小红书笔记，结果发现要么需要配置各种API密钥，要么得用Playwright模拟浏览器操作，Token消耗还特别大？

OpenCLI彻底解决了这个问题。它的原理很巧妙：通过一个Chrome扩展，复用你已经登录的浏览器会话，然后用命令行直接操作。**不需要API Key，不消耗LLM Token，输出是干净的结构化数据**。

### 安装三步走

```bash
# 1. 全局安装
npm install -g @jackwener/opencli

# 2. 验证daemon
opencli doctor

# 3. Chrome加载扩展后再次验证
opencli doctor
```

装完之后，像操作B站就简单了：

```bash
opencli bilibili me       # 看自己的B站信息
opencli bilibili hot      # 热门视频
opencli bilibili comments # 评论列表
opencli bilibili subtitle # 字幕提取
```

它支持的站点远不止B站：知乎、小红书、Reddit、Twitter、HackerNews……一共90多个适配器。对于我们这种用AI Agent做自动化的人来说，这工具简直是降本神器——以前用Playwright+LLM做一次网页操作可能消耗几千Token，现在一个命令搞定，Token消耗为零。

不过有个小坑：`opencli doctor`显示daemon正常但extension没连上，需要手动去Chrome的`chrome://extensions/`页面加载扩展才行。

## 🐛 DeepSeek回退模型踩坑：一个协议选择引发的血案

再来说一个排查了半天的Bug。

我的Hermes Agent配置了DeepSeek作为回退模型（主模型GLM-5.1限流时自动切换）。之前一直用Anthropic协议（`base_url: https://api.deepseek.com/anthropic`），结果发现一个诡异的问题：**主模型一触发限流，回退到DeepSeek就报HTTP 400错误**。

排查后发现根因：Hermes的`anthropic_adapter.py`在转换消息格式时，会**把非Anthropic格式的thinking blocks全部剥离**。这意味着当GLM-5.1的对话历史（包含thinking blocks）传给DeepSeek的Anthropic端点时，消息格式被破坏了。

**解决方案出乎意料地简单**：把`base_url`从`https://api.deepseek.com/anthropic`改成`https://api.deepseek.com`，走OpenAI协议就行了。一个URL后缀的区别，但背后的协议处理逻辑完全不同。

这个坑的教训是：**当API服务同时支持多种协议时，要特别注意中间adapter层的行为**。协议选择不只是URL的区别，还影响消息格式的转换逻辑。

## 📡 Follow-Builders：免费追踪AI圈最新动态

最后一个好东西——**Follow-Builders**（GitHub: zarazhangrui/follow-builders）。

这个项目聚合了25个AI领域的大V推特账号、6个顶级AI播客和2个技术博客的内容。关键是：**它把所有feed数据以JSON格式公开在GitHub上**，不需要配置任何API：

- `feed-x.json` — 25个X账号的最新动态
- `feed-podcasts.json` — 6个播客更新
- `feed-blogs.json` — 2个博客文章

我的计划是：每天定时拉取这些JSON，用LLM生成中文摘要，然后推送到公众号和飞书。相当于一个免费的"AI前沿日报"生成器，而且完全不需要X API或YouTube字幕抓取。

追踪的播客质量都很高：Latent Space、Training Data、No Priors、Unsupervised Learning等，都是AI从业者必听的节目。
